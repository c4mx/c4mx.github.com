My investigation

In the first time, I checked the source code of the html page, and I knew that the password should be in format of CTF{([0-9a-zA-Z_@!?-]+)}. The part between the braces is then passed to function x(x). If the function returns true, the state of the safe will become ‘granted’, otherwise it’s ‘denied’.

I decided to debug this bullshit. I put some breakpoint inside funciton h, I get the value of s when h is called, it turns out it’s the source code of the function x itself!

How could that be possible? I know the code socpe in JavaScript maybe tricky, but here it’s hard to believe what I see. At some moment, I suspect that the paramter x is not the same character with the function x(), especially check it in my editor, but I didn’t go into it seriously.

So if it’s the source code of function x is passed, and to debug I modified the code of x, that will surely not work. This is a kind of anti-tamper feature, cool!

At this point, I don’t know how to continue my test. I don’t understand either what with(source) is for.

Solution

Pitfalls

There are several pitfalls in this challenge:

  1. The paramer x and the function x are two different letters. The fomer is a cyrillic alphabet. This surely creates confusion.
  2. If the debugger loop is commented directly, the variable a will be undefined instead of 1000 when the function h is called.
  3. If you add console.log(source) somewhere in the code and debug it, the execution will go into infinit loop, as when c(source, x) is called, the regex source doesn’t have length property, so source.length will give a undefined, thus a infinite loop.
  4. If you comment the with statement, add your own c(source, x) call, the execution will go into infinite loop. It’s beacause the source is regex, as shown in number 3, the function will go into infinite loop.

The with statement is to extends the scope chain for a statement. When searching the source in return eval('eval(c(source,x))'), the property source which in with expression is first found instead of regxex source. The ‘source’ propery of a regex in the with expression is not the one defined as a regex in the code !

How to approche it

In fact, analysing the h and c functions, you will find:

  • h is a hashing function or key seed function that giving a key seed, it will generate a new key from it.
  • c is a simple XOR encryption function.

One key step is to print the final return value (c) in the function c. You will get two outputs, the first output is critical to know:

This is a valid js expression which will be passed to eval. Now there are two important points:

  1. The x in h(x) refers to the parameter of function x, that’s our password.
  2. If the expression is evaluated to True, then function x will return True.

In other word:

password == weird_string XOR hash(password) should be evaluated to True.

Finding the password

How do we find the right password? Luckily there are three things we can exploit:

  • We know the password should have characters only from 0-9a-zA-Z_@!?-
  • We know the length of the hash is 4-byte length, in total, 2^32 possiblities.
  • We know, in the xor function ‘x’, the hash is xored with password in module of hash’s length, which is 4.

Considering that, we can brute-force each byte of the hash (b0b1b2b3):

  • Each byte hash 2^8=256 possiblities
  • A right candidate value should make each xored byte in the 0-9a-zA-Z_@!?- set.

So I create a python script for it (check it on my github).

The final flag is: CTF{_N3x7-v3R51ON-h45-AnTI-4NTi-ant1-D3bUg_}
